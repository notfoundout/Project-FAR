# FARM Requirement Routing

## Purpose

This document defines how FARM routes downstream requirements to the correct Project FAR layer.

It does not decide the substantive content of any stable layer.

---

## Requirement

A requirement is a recorded need for a framework artifact, correction, extension, clarification, proof, example, or operational capability.

---

## Required Fields

A routed requirement should identify:

- requirement identifier;
- source artifact;
- source context;
- requirement statement;
- affected layer;
- proposed target document;
- reason for routing;
- urgency or blocking status;
- accepted, rejected, deferred, or unresolved status.

---

## Target Layers

Requirements may be routed to:

- FARA, when representational architecture is affected;
- FAR, when investigation methodology is affected;
- FARO, when operational procedures are affected;
- FARE, when mathematical support is required;
- FARM, when cross-framework governance is affected;
- examples, when concrete artifact demonstration is required;
- project governance, when only status or milestone recording is affected.

---

## Routing Rule

A requirement shall not be routed to a stable framework layer unless the routing reason explains why the issue cannot be resolved downstream.

---

## Boundary Rule

Routing a requirement does not authorize a change.

It only identifies where review should occur.
