# FAR Commercial Validation Protocol v1

## Purpose

This protocol tests which initial FAR commercial wedge produces the strongest evidence of operational value. It does not test or establish the universal Project FAR research thesis.

## Candidate wedges

1. Release assurance before deploying an agent update.
2. Incident reconstruction after an unexplained agent action.
3. Procurement evidence during enterprise review.
4. Instrumentation sufficiency when existing artifacts cannot support required claims.

## Eligible testers

Prioritize participants who can evaluate or influence AI engineering, security, governance, procurement, incident response, or enterprise implementation decisions. General-interest visitors may submit feedback, but their responses must be segmented from qualified buyer evidence.

## Standard task

1. Review the flagship treasury-release scenario.
2. Run the FAR investigation.
3. Inspect the release decision, structural regressions, declared unknowns, rule findings, and evidence fingerprint.
4. Submit the structured validation form.
5. Do not upload confidential, proprietary, personal, regulated, credential-bearing, or production-sensitive artifacts.

## Evidence hierarchy

Rank signals in this order:

1. Payment.
2. Signed pilot or purchase commitment.
3. Authorized access to real execution artifacts.
4. Named budget owner and active buying event.
5. Scheduled design partnership.
6. Concrete recent incident or blocked release.
7. Detailed workflow and tool description.
8. Structured site feedback.
9. Generic praise.

The site form produces level-eight evidence. It is useful for segmentation and hypothesis selection but does not by itself validate willingness to pay.

## Primary measures

- Use case selected by role and organization type.
- Agreement with the release action.
- Clarity score.
- Trust score.
- Willingness to provide authorized artifacts.
- Whether a budget owner is identifiable.
- Design-partner interest.
- Conversion to a scheduled conversation or pilot.

## Source attribution

Tester invitations should use a controlled `source` query parameter, for example:

`/?source=security-founder-outreach-01`

Allowed characters are letters, numbers, dots, underscores, colons, and hyphens. Do not encode personal information in the source value.

## Privacy boundary

The validation endpoint intentionally does not collect:

- uploaded package contents;
- uploaded filenames;
- free-text feedback;
- IP addresses;
- browser fingerprints;
- user-agent strings.

The structured feedback record may contain an optional contact email supplied with consent. Events are written to application logs as canonical JSON for the initial controlled study. Production-scale validation requires durable access-controlled storage, retention limits, deletion procedures, and an updated privacy notice.

## Interpretation

Do not combine all responses into a single popularity score. Segment by tester competence, role, organization, purchasing authority, artifact access, and active trigger. A smaller number of high-quality commercial signals outweighs a larger number of unqualified positive ratings.

## Stop conditions

Pause external testing if:

- uploaded content or filenames appear in feedback logs;
- the form records data without consent;
- sample-specific facts leak into uploaded analyses;
- qualified testers systematically misunderstand what FAR verifies;
- the site implies certification, complete safety, hidden-reasoning access, or external truth.
