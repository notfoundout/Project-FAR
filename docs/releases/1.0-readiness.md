# Project FAR 1.0 readiness gate

Project FAR will not publish another intermediate release after `0.4.0`. The next public release is `1.0.0`.

## Release claim

Project FAR 1.0 can ingest supported external reasoning artifacts, preserve source provenance and declared uncertainty, compile them into deterministic decision packages, apply explicit evidence-sufficiency rules, and produce auditable reports and evidence manifests.

## Required before tagging 1.0.0

- Candidate 001 and Candidate 002 findings are recorded with evidence hashes and claim boundaries.
- Installation from a clean checkout succeeds on the supported Python version.
- Canonical CLIs and package names are documented and stable.
- FAR, FARA, FARE, FARM, and FARO boundaries are consistent across the repository.
- Theory, verification core, commercial applications, research experiments, and archived work are visibly separated.
- Schemas, examples, documentation, and executable behavior agree.
- One golden external-trace example reproduces deterministically from documented commands.
- The cumulative changes since `0.4.0` are summarized in release notes.
- The complete repository validation and validator-assurance suites pass on the exact release commit.
- A frozen 1.0 claim boundary states what the release does not prove.

## 1.0 nonclaims

Project FAR 1.0 does not claim:

- unrestricted universality across all reasoning systems;
- complete access to hidden or tacit reasoning;
- automatic factual verification of every trace statement;
- population-level failure-detection accuracy from two external candidates;
- independent organizational replication;
- regulatory certification;
- that `unverifiable` means safe, correct, or failure-free.

## Release sequence

1. Record Candidate 002 findings.
2. Consolidate package, CLI, schema, and framework documentation.
3. Create the cumulative `0.4.0` to `1.0.0` release notes.
4. Add a clean-install golden reproduction workflow.
5. Run the full release validation matrix on the exact candidate commit.
6. Tag and publish `1.0.0` only after every gate is satisfied.
