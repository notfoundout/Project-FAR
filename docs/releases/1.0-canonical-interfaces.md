# Project FAR 1.0 Canonical Interfaces

This document defines the public interface surface intended for Project FAR 1.0.0. It does not change the mathematical claim boundary or promote experimental components into supported interfaces.

## Framework roles

- **FAR**: the formal representation and reasoning framework.
- **FARA**: assurance and validation of FAR implementations and artifacts.
- **FARE**: evaluation of reasoning artifacts and evidence.
- **FARM**: mechanization and machine-checked formal results.
- **FARO**: operational orchestration around supported FAR workflows.

These names describe distinct roles. Product packages may implement one or more roles, but must not redefine them.

## Canonical command-line interfaces

- `far-decision`: validate and adjudicate a FAR decision package.
- `far-reasoning-regression`: compare baseline and candidate decision corpora and emit deterministic regression evidence.
- `far-external-trace`: compile supported external trace formats into FAR decision packages.

Command names above are the supported 1.0 CLI surface. Experimental scripts under `research/` are reproducibility assets, not general-purpose public commands.

## Canonical schemas

- `far-decision-package/0.1`: decision package input.
- `far-external-trace/0.1`: normalized external trace representation.
- `far-reasoning-regression/0.1`: regression comparison output.
- `far-evidence-bundle/0.1`: deterministic evidence manifest.

Schema identifiers remain versioned independently from the Project FAR release number. A Project FAR 1.0 release does not imply every schema identifier must be renamed to `1.0`.

## Supported result states

Adjudication returns one of:

- `justified`
- `unsupported`
- `underdetermined`
- `unverifiable`

`unverifiable` is a first-class result. Missing or incomplete evidence must not be converted into a negative factual claim.

## Stability boundary

The following are intended to remain stable through the 1.x line unless a documented compatibility exception is required:

- canonical CLI names;
- top-level schema identifiers;
- four-state adjudication semantics;
- provenance classes and source-reference requirements;
- deterministic report and manifest behavior.

Internal Python module paths, research scripts, experimental adapters, and unreleased service infrastructure are not part of this compatibility promise.
